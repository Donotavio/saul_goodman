# i18n Rules

## Formato Chrome
- `_locales/<locale>/messages.json`
- Keys descritivos (nunca genericos)
- Nunca colocar conteudo dinamico em `data-i18n-html`
- Acessar via `chrome.i18n.getMessage('keyName')`

## 14 Locales suportados
- en_US (base), pt_BR, es_419, ar, bn, de, fr, hi, it, ru, tr, ur, zh_CN
- RTL: ar (arabe), ur (urdu) — testar layout invertido

## Tres sistemas de i18n
1. Chrome extension: `_locales/<locale>/messages.json`
2. VS Code extension: `package.nls.<locale>.json` + `vscode-extension/_locales/`
3. Site: `site/_locales/` (copias gitignored de `_locales/`)

## Workflow apos mudanca de texto
1. Adicionar/atualizar key em `_locales/en_US/messages.json`
2. Rodar `npm run i18n:sync` para propagar
3. Rodar `npm run i18n:check` para validar completude
4. Se necessario: `npm run i18n:repair-locale` para correcao via LLM
5. Para VS Code: atualizar `package.nls.*.json` manualmente

## Regras
- Toda string visivel ao usuario DEVE ter i18n key
- Nunca hardcodar texto em HTML/TS
- Manter consistencia de tom entre locales
- Testar RTL apos mudancas em ar/ur
